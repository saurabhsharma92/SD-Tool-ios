Distributed Systems Fundamentals

Introduction

A distributed system is a collection of independent computers that appear to users as a single coherent system. From social networks serving billions of users to banking platforms processing millions of transactions, nearly every modern application is a distributed system at its core.

Working with distributed systems introduces challenges that do not exist in single-machine computing: partial failures, network unreliability, clock skew, and the impossibility of global agreement without coordination. This article covers the fundamental building blocks -- consensus, leader election, distributed transactions, locking, gossip protocols, failure detection, clock synchronization, and more. These concepts appear in virtually every system design interview.

Why Distributed Systems

The Four Motivations

Motivation What It Means Example
Scalability Handle more load by adding machines A single database server handles 10K queries/sec; a distributed cluster handles 1M
Availability Continue operating despite failures If one data center goes down, another serves traffic
Fault tolerance Tolerate partial failures gracefully A failed node does not bring down the entire system
Geographic distribution Serve users from nearby locations Users in Tokyo hit a Tokyo data center, not one in Virginia

The Fundamental Challenges

  1. Partial failures: In a single machine, either everything works or nothing does. In a distributed system, some nodes can fail while others continue. You must detect and handle partial failures.
  2. No global clock: Each machine has its own clock, and clocks drift. You cannot rely on timestamps to order events across machines.
  3. Network unreliability: Messages can be lost, delayed, duplicated, or delivered out of order. A node that does not respond might be crashed, slow, or separated by a network partition.
  4. Concurrency: Multiple nodes process requests simultaneously. Without coordination, they can make conflicting decisions.

The core difficulty of distributed systems is not any single one of these challenges -- it is that they all happen simultaneously and interact with each other. A partially failed node with a drifting clock on the far side of a network partition is a realistic scenario, not an edge case.

Consensus Algorithms

The Problem

Multiple nodes need to agree on a single value (e.g., who is the leader, what is the next transaction to commit, which configuration is active). This must work even if some nodes crash or messages are lost.

The FLP impossibility result (Fischer, Lynch, Paterson, 1985) proves that deterministic consensus is impossible in an asynchronous system where even one node can crash. Practical algorithms work around this by using timeouts (introducing synchrony assumptions) or randomization.

Paxos

Paxos, proposed by Leslie Lamport, is the foundational consensus algorithm. It is famously difficult to understand, but the core idea is straightforward.

Roles:

  • Proposer: Proposes a value to be agreed upon
  • Acceptor: Votes on proposals. A majority of acceptors must agree.
  • Learner: Learns the decided value once consensus is reached

Simplified two-phase protocol:

Phase 1 (Prepare):

  1. Proposer picks a unique proposal number N (higher than any it has used before)
  2. Proposer sends Prepare(N) to a majority of acceptors
  3. Each acceptor responds with a promise: "I will not accept any proposal numbered less than N." If it has already accepted a previous proposal, it includes that value.

Phase 2 (Accept):

  1. If the proposer receives promises from a majority, it sends Accept(N, value) where value is either the highest-numbered previously accepted value or the proposer's own value
  2. Each acceptor accepts the proposal unless it has already promised to a higher number
  3. Once a majority of acceptors accept, the value is decided

Why Paxos is hard in practice:

  • Multiple proposers can conflict, causing livelock (each keeps proposing higher numbers)
  • Multi-Paxos (for a sequence of values) adds significant complexity
  • Implementation details (log management, snapshotting, membership changes) are underspecified

Raft

Raft was designed by Diego Ongaro as a more understandable alternative to Paxos. It achieves the same safety guarantees but is structured around a strong leader, making it easier to implement and reason about.

Three sub-problems:

  1. Leader election
  2. Log replication
  3. Safety

How it works:

  1. All nodes start as followers. They expect periodic heartbeats from a leader.
  2. If a follower receives no heartbeat within a randomized timeout (e.g., 150-300ms), it becomes a candidate and starts an election.
  3. The candidate increments its term number (a logical clock), votes for itself, and requests votes from all other nodes.
  4. Each node votes for at most one candidate per term (first-come-first-served).
  5. If the candidate receives votes from a majority, it becomes the leader and starts sending heartbeats.
  6. If two candidates split the vote (neither gets a majority), both time out and start a new election with incremented terms. Randomized timeouts make repeated splits unlikely.

Raft's leader election is one of the most commonly asked distributed systems topics. The key insight to communicate is: randomized election timeouts prevent split votes from repeating indefinitely. Without randomization, two candidates could keep tying forever.

Log Replication

Once elected, the leader handles all client requests:

  1. Client sends a command to the leader
  2. Leader appends the command to its log as a new entry
  3. Leader sends AppendEntries RPCs to all followers
  4. Followers append the entry to their logs and acknowledge
  5. Once a majority of nodes have the entry, the leader commits it and applies it to its state machine
  6. The leader notifies the client of success
  7. Followers learn of the commit in subsequent heartbeats and apply the entry

Safety guarantee: If an entry is committed, it is guaranteed to be present in the log of every future leader. Raft ensures this by requiring candidates to have all committed entries in their log before they can win an election.

Systems using Raft: etcd (Kubernetes), Consul, CockroachDB, TiKV.

Leader Election

Beyond Raft's built-in election, there are several classical leader election algorithms.

Bully Algorithm

Used when nodes have unique, comparable IDs (e.g., process IDs).

  1. When a node detects the leader is down, it sends an election message to all nodes with higher IDs
  2. If any higher-ID node responds with alive, the initiator stops (it defers to the higher node)
  3. The highest-ID node that does not receive a response from any higher node declares itself leader
  4. The new leader broadcasts a coordinator message to all nodes

Problems: Assumes reliable failure detection. In practice, slow nodes can be mistaken for dead nodes, leading to unnecessary elections.

Ring Algorithm

Nodes are arranged in a logical ring. When a node detects the leader is down:

  1. It sends an election message around the ring (to its successor)
  2. Each node appends its ID to the message and forwards it
  3. When the message returns to the initiator, it picks the node with the highest ID as the leader
  4. A coordinator message is sent around the ring announcing the winner

ZooKeeper-Based Leader Election

ZooKeeper provides a practical and reliable leader election mechanism using ephemeral sequential nodes:

  1. Each candidate creates an ephemeral sequential znode under a designated path (e.g., /election/node-000001)
  2. The candidate with the lowest sequence number is the leader
  3. All other candidates set a watch on the node with the next-lower sequence number
  4. When the leader dies, its ephemeral node is automatically deleted
  5. The next candidate's watch fires, and it checks if it now has the lowest sequence number
  6. If so, it becomes the new leader

Why this works well:

  • Ephemeral nodes automatically disappear when the session ends (crash detection for free)
  • Sequential nodes provide a natural ordering
  • Watching only the previous node prevents a "herd effect" where all candidates react to every change

Distributed Transactions

Two-Phase Commit (2PC)

2PC coordinates a transaction across multiple participants (databases, services) so that all commit or all abort.

Phase 1 (Prepare/Voting):

  1. The coordinator sends a PREPARE message to all participants
  2. Each participant executes the transaction up to the point of committing (acquires locks, writes to WAL)
  3. Each participant votes YES (ready to commit) or NO (must abort)

Phase 2 (Commit/Abort):

  • If ALL participants voted YES, the coordinator sends COMMIT to all
  • If ANY participant voted NO, the coordinator sends ABORT to all
  • Participants execute the decision and release locks

The Blocking Problem with 2PC

2PC has a critical flaw: if the coordinator crashes after Phase 1 but before sending the Phase 2 decision, participants that voted YES are stuck. They hold locks and cannot decide on their own whether to commit or abort. They must wait for the coordinator to recover.

This means:

  • Resources are locked indefinitely
  • Other transactions waiting for those locks are also blocked
  • The system's availability is reduced to the coordinator's availability

2PC is a blocking protocol. If the coordinator fails after collecting votes but before broadcasting the decision, participants are in an uncertain state. They hold locks and wait. In practice, this makes 2PC unsuitable for high-availability systems. Most modern distributed systems use saga patterns or avoid distributed transactions entirely.

Three-Phase Commit (3PC)

3PC adds an intermediate phase to avoid the blocking problem:

  1. Phase 1 (CanCommit): Coordinator asks participants if they can commit
  2. Phase 2 (PreCommit): If all say yes, coordinator sends precommit (participants prepare but do not commit yet; they know everyone agreed)
  3. Phase 3 (DoCommit): Coordinator sends the final commit

The key improvement: If the coordinator fails after Phase 2, participants know that everyone voted yes (because the coordinator only proceeds to Phase 2 if all agreed). They can safely proceed to commit on their own after a timeout.

Limitation: 3PC assumes that network partitions do not occur. If a partition splits participants, different sides might make different decisions. This makes 3PC unsuitable for systems where partitions are possible -- which is most distributed systems.

Saga Pattern as an Alternative

Because 2PC and 3PC both have significant drawbacks, modern distributed systems often use the saga pattern instead. Rather than coordinating a single atomic distributed transaction, a saga breaks the operation into a sequence of local transactions, each with a compensating action for rollback. (See the Microservices Architecture article for details.)

Distributed Locking

Why Distributed Locks Are Hard

A lock on a single machine is straightforward: use a mutex. In a distributed system, multiple processes on different machines need to coordinate access to a shared resource. This requires a lock service that all processes can access.

Requirements for a correct distributed lock:

  1. Mutual exclusion: At most one client holds the lock at any time
  2. Deadlock freedom: Eventually, a lock is always released (even if the holder crashes)
  3. Fault tolerance: The lock service survives node failures

Redis Redlock Algorithm

Martin Kleppmann famously criticized Redlock, but it remains widely discussed in interviews. Here is how it works:

Setup: N independent Redis instances (typically 5), running on separate machines.

Acquiring a lock:

  1. Client records the current time (T1)
  2. Client tries to acquire the lock on each Redis instance sequentially, with a short timeout per attempt
  3. The lock is acquired if the client successfully locked a majority (N/2 + 1) of instances
  4. The effective lock validity time is the original TTL minus the time elapsed (T2 - T1)
  5. If the client fails to lock a majority, it releases any locks it did acquire

Releasing a lock:

  • Client sends DEL to all N instances

Concerns (per Kleppmann):

  • Relies on clock synchronization across Redis instances (clocks can jump)
  • A GC pause or network delay can cause a client to use a lock after it has expired
  • Not safe without fencing tokens (see below)

ZooKeeper Ephemeral Nodes

ZooKeeper provides a more principled distributed lock:

  1. Client creates an ephemeral sequential node under /locks/resource-name
  2. If the client's node has the lowest sequence number, it holds the lock
  3. Otherwise, the client watches the node with the next-lower sequence number
  4. When that node is deleted (lock released or holder crashed), the client rechecks
  5. If the client disconnects (crash, network failure), the ephemeral node is automatically deleted, releasing the lock

Advantages over Redis:

  • Ephemeral nodes provide automatic lock release on crash
  • ZooKeeper is built on consensus (ZAB protocol), so it is consistent
  • No reliance on clock synchronization

Fencing Tokens

Even with distributed locks, a problem remains: a client might acquire a lock, experience a GC pause or network delay, and by the time it uses the lock, the lock has expired and another client holds it. Both clients now think they have the lock.

Solution: Fencing tokens. The lock service issues a monotonically increasing token with each lock acquisition. The resource being protected (e.g., a database, a file store) rejects any request with a token lower than the highest token it has seen.

In interviews, always mention fencing tokens when discussing distributed locks. The follow-up question is often "what happens if the lock holder pauses?" Without fencing, the answer is "data corruption." With fencing, the answer is "the stale holder's writes are rejected."

Algorithm:

  1. Every T seconds (e.g., T=1), each node selects a random peer
  2. The node sends its current state (or a digest of it) to that peer
  3. The peer merges the received state with its own (keeping newer information)
  4. The peer may send back any information the sender was missing

Convergence: With N nodes and each node gossiping once per round, information reaches all nodes in O(log N) rounds. This is mathematically similar to how epidemics spread -- hence the name.

Use Cases

Membership and failure detection: Each node maintains a list of known members with heartbeat counters. When a node gossips, it increments its own counter. If a node's counter has not increased after several rounds, it is suspected of being down. (Cassandra uses this approach.)

Data dissemination: Spread lightweight data (cluster metadata, configuration changes, routing tables) to all nodes without a central coordinator.

Aggregate computation: Compute distributed averages, sums, or other aggregates by having nodes exchange and merge partial results.

Heartbeat Mechanisms

Basic Heartbeats

A heartbeat is a periodic signal sent by a node to indicate it is alive. If a monitor node does not receive heartbeats from a remote node for a configurable period, it declares the remote node as failed.

Parameters:

  • Heartbeat interval: How often heartbeats are sent (e.g., every 1 second)
  • Timeout threshold: How many missed heartbeats before declaring failure (e.g., 3 missed = 3 seconds)

Tradeoff: Short intervals detect failures faster but generate more network traffic and increase false positives (a slow node might be declared dead). Long intervals reduce traffic but delay failure detection.

Phi Accrual Failure Detector

Traditional heartbeat monitors use a fixed timeout: miss 3 heartbeats and you are declared dead. This binary approach does not adapt to varying network conditions.

The phi accrual failure detector (used by Akka and Cassandra) provides a continuous suspicion level (phi) rather than a binary alive/dead judgment:

  1. Track the inter-arrival times of heartbeats (build a statistical distribution)
  2. When a heartbeat is late, compute phi based on how unlikely the current delay is given the historical distribution
  3. Higher phi = more likely the node has failed
  4. The application chooses a phi threshold for declaring failure (e.g., phi > 8 means very likely dead)

Advantages:

  • Adapts to network conditions automatically
  • Provides a suspicion level rather than a binary judgment
  • Different applications can use different phi thresholds depending on their sensitivity

In practice, the phi accrual detector reduces false positives compared to fixed-timeout heartbeats, especially in environments with variable network latency (cloud environments, cross-datacenter communication). Cassandra uses phi = 8 as its default threshold.

Clock Synchronization

The Problem

Distributed systems need to order events, but each node has its own physical clock. These clocks drift due to hardware imperfections, temperature changes, and other factors. Two nodes can disagree on the current time by milliseconds to seconds.

NTP (Network Time Protocol)

NTP synchronizes clocks by querying time servers. A client sends a request to an NTP server, and by measuring the round-trip time, estimates the clock offset.

Limitations:

  • Accuracy is typically 1-10ms over the internet, 0.1-1ms on a LAN
  • Cannot guarantee bounded clock skew
  • Clocks can jump backward (NTP adjustments), which can break software that assumes time always moves forward
  • Not precise enough for ordering events in fast-paced distributed systems

Lamport Clocks

A logical clock that provides a partial ordering of events without relying on physical time.

Rules:

  1. Each node maintains a counter C, initialized to 0
  2. Before each local event, increment C: C = C + 1
  3. When sending a message, include the current C value
  4. When receiving a message with timestamp T: C = max(C, T) + 1

Property: If event A happened before event B (causally), then C(A) < C(B). But the converse is NOT true: C(A) < C(B) does not mean A happened before B. Concurrent events may have any ordering of timestamps.

Vector Clocks

Vector clocks extend Lamport clocks to detect concurrency. Each node maintains a vector of counters -- one per node in the system.

Rules:

  1. Node i increments VC[i] before each local event
  2. When sending a message, include the full vector clock
  3. When receiving a message with vector VCm: for each entry j, set VC[j] = max(VC[j], VCm[j]), then increment VC[i]

Comparison:

  • VC(A) < VC(B) (every element of A <= corresponding element of B, and at least one is strictly less): A causally precedes B
  • VC(A) || VC(B) (neither dominates): A and B are concurrent

Limitation: Vector clock size grows with the number of nodes. For large clusters, this overhead becomes significant. Some systems use bounded vector clocks (prune old entries) at the cost of precision.

Hybrid Logical Clocks (HLC)

HLCs combine physical and logical clocks. They maintain a physical component (close to wall-clock time) and a logical component (for ordering events within the same physical time).

Advantages:

  • Bounded size (constant, unlike vector clocks)
  • Close to physical time (useful for TTL, timeouts, human-readable timestamps)
  • Captures causality like Lamport clocks
  • Used by CockroachDB and other modern distributed databases

When discussing event ordering in interviews, start with the problem (physical clocks drift, cannot rely on wall time), mention Lamport clocks (simple but cannot detect concurrency), then vector clocks (detect concurrency but grow with cluster size), and optionally mention HLCs as a practical compromise. This progression shows deep understanding.

Split-Brain Problem

What Is Split-Brain?

Split-brain occurs when a network partition divides a cluster into two or more groups, and each group believes it is the only active group. Both sides may elect their own leader and accept writes, leading to divergent state.

Example:

  • A 5-node cluster is split into a group of 3 and a group of 2
  • Both groups elect a leader (if using a simple algorithm that does not require a majority)
  • Both leaders accept writes
  • When the partition heals, the two leaders have conflicting data

Prevention Strategies

Majority quorum: A leader can only be elected (and operations can only proceed) if the node has support from a majority (N/2 + 1) of nodes. In a 5-node cluster with a 3-2 split, only the group of 3 can elect a leader. The group of 2 becomes read-only or unavailable.

Fencing: When a new leader is elected, it "fences off" the old leader by revoking its access to shared resources. This can be done by changing access tokens, shutting down the old leader's network, or using STONITH (Shoot The Other Node In The Head) in hardware clusters.

Epoch/term numbers: Each leader election produces a new epoch or term number (as in Raft). Any operation with an old epoch number is rejected. This prevents a stale leader from making changes after being deposed.

Split-brain can cause data corruption that is extremely difficult to fix after the fact. Always design for it. In interviews, if you describe a system with replication and leader election, the interviewer may ask "what about split-brain?" The answer is quorum-based election (majority required) and fencing tokens.

Hinted Handoff

Handling Temporary Failures

When a node is temporarily unavailable (network blip, rolling upgrade, brief crash), the system still needs to accept writes destined for that node.

How hinted handoff works:

  1. A write request arrives for Node C, but Node C is currently unreachable
  2. Instead of failing the write, Node A (any available node) accepts the write and stores it locally with a "hint" indicating it belongs to Node C
  3. Node A periodically checks if Node C is back
  4. When Node C recovers, Node A forwards the stored writes to Node C
  5. Node A deletes the hints after successful delivery

Used by: Cassandra, Riak, DynamoDB (original Dynamo paper).

Tradeoff: Hinted handoff improves write availability during temporary failures but does not help with permanent failures (if Node C never comes back, the hints pile up on Node A). It also weakens consistency -- reads from Node C after recovery but before hint delivery will miss the hinted writes.

Anti-Entropy with Merkle Trees

The Problem

Over time, replicas can diverge due to missed updates, failed hinted handoffs, or bitrot. Anti-entropy is the process of comparing replicas and synchronizing them.

Naive approach: Compare every key-value pair between two replicas. This is O(n) in data size -- prohibitively expensive for large datasets.

Merkle Trees

A Merkle tree (hash tree) is a binary tree where:

  • Leaf nodes contain the hash of a data block (a key-value pair or a range of keys)
  • Internal nodes contain the hash of the concatenation of their children's hashes
  • The root is a single hash that summarizes the entire dataset

Comparing replicas with Merkle trees:

  1. Each replica maintains a Merkle tree over its data
  2. To compare, exchange root hashes. If they match, the replicas are identical. Done.
  3. If they differ, compare the children of the root. If the left children match but the right children differ, only traverse the right subtree.
  4. Continue recursively until you find the specific leaf nodes (data blocks) that differ
  5. Synchronize only the differing data

Complexity: O(log N) comparisons to find differences, rather than O(N) with a full scan.

Used by: Cassandra (for repair operations), Dynamo, IPFS, Git (commit hashes form a DAG, not exactly a Merkle tree, but the principle is the same).

Merkle trees are an optimization for detecting differences, not for storing data. The actual data remains in the database; the Merkle tree is a parallel structure that is rebuilt or incrementally updated as data changes. Cassandra builds Merkle trees on-demand during repair operations.

MapReduce

The Paradigm

MapReduce is a programming model for processing large datasets in parallel across a cluster. It abstracts away the complexity of distributed execution, fault tolerance, and data partitioning.

Three phases:

  1. Map: Apply a function to each input record, emitting zero or more key-value pairs
  2. Shuffle: Group all emitted values by key (this is handled by the framework)
  3. Reduce: For each unique key, combine all associated values into a final result

Word Count Example

Input: A large collection of text documents

Map function: For each word in a document, emit (word, 1)

Limitations of MapReduce

Limitation Impact
Batch only Not suitable for real-time or streaming workloads
Disk-heavy Intermediate results written to disk between map and reduce; slow
Two-stage model Complex computations require chaining multiple MapReduce jobs
High latency Minutes to hours, even for simple computations

MapReduce is conceptually important but technologically dated. In interviews, use it to explain the general idea of distributed data processing (partition, process in parallel, aggregate). But for system design, propose Spark or Flink for batch/stream processing. Saying "I'd use MapReduce" for a modern system can signal outdated thinking.

Putting It All Together

How These Concepts Interact

Real distributed systems combine these building blocks:

Cassandra uses gossip for membership, hinted handoff for temporary failures, Merkle trees for anti-entropy, vector clocks (originally) or LWW timestamps for conflict resolution, and tunable quorums for consistency.

etcd uses Raft for consensus and leader election, distributed locking for coordination, and linearizable reads for strong consistency.

ZooKeeper uses ZAB (similar to Raft) for consensus, ephemeral nodes for leader election and locking, and watches for change notification.

Spanner uses Paxos for consensus, TrueTime (GPS + atomic clocks) for clock synchronization, 2PC for cross-shard transactions (made practical by TrueTime's bounded uncertainty), and Merkle-like structures for data integrity.

Vector clock Per-node counters; detects both causality and concurrency; size grows with cluster
HLC Hybrid logical clock; bounded size, close to wall time, captures causality; used by CockroachDB
Split-brain Partition causes dual leaders; prevent with majority quorum + fencing
Hinted handoff Write to available node with hint; forward to correct node on recovery
Merkle tree Hash tree for efficient replica comparison; O(log N) to find differences
MapReduce Map (transform) -> Shuffle (group by key) -> Reduce (aggregate); batch processing paradigm

In system design interviews, do not just name these concepts -- explain the tradeoff. "I'd use Raft for consensus because it is easier to implement and reason about than Paxos, though both provide the same safety guarantees." "I'd use gossip for membership because it is scalable and fault-tolerant, but it provides only eventual consistency for membership information." Showing that you understand WHY to use each tool, not just WHAT it does, is what distinguishes senior candidates.
